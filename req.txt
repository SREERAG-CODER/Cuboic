1. Introduction
1.1 Purpose
This document describes the functional and non-functional requirements.
The system enables customers to order food via QR code, allows restaurant staff to manage and assign delivery robots, and provides restaurant owners with a mobile dashboard to monitor operations.
1.2 Scope
The system consists of:
A 
Customer Web Application
A 
Staff Web Dashboard
A 
Robot Management Interface
A 
Owner Mobile Application
A Backend API Server with Database
The system will:
Allow QR-based food ordering
Enable order status tracking
Allow staff to assign robots and cabinets
Simulate cabinet delivery logic
Provide operational insights to the owner
This is an MVP-focused implementation for public deployment.

3. Functional Requirements
3.1 Customer Web Application
FR-1: QR Menu Access
The system shall allow customers to access menu via QR code.
QR must include restaurant_id and table_id.
FR-2: View Menu
System shall display:
Item name
Price
Category
FR-3: Cart Management
Add items
Remove items
Update quantity
View total price
FR-4: Checkout
Proceed to payment page
Process payment using payment gateway
Generate order upon successful payment
FR-5: Order Status Tracking
Customer shall view:
Order Received
Preparing
Ready
Assigned to Robot
Delivering
Delivered
3.2 Staff Web Dashboard
FR-7: View Orders
Staff shall view:
Order ID
Table number
Items
Status
Assigned robot
FR-8: Update Order Status
Staff shall update order stages:
Received
Preparing
Ready
FR-9: Robot & Cabinet Assignment 
The system shall allow staff to assign:
One robot to multiple orders simultaneously.
One order to one or more cabinets within the same robot.
Rules:
A robot may carry up to 3 cabinet assignments at a time.
Each cabinet (C1, C2, C3) can:
Serve one table only.
Contain items from one order.
A single order may occupy:
One cabinet (small order)
Multiple cabinets (large order)
FR-10: Mark Delivery
Staff/Robot shall mark order as delivered.
System shall:
Free cabinet(s) assigned to the delivered order.
If no cabinets remain occupied, set robot status to â€œIdleâ€.
Otherwise, continue delivery to next table in queue.
FR-11: Multi-Table Delivery Handling
The system shall:
Allow robot to carry orders for up to 3 different tables in one trip.
Track cabinet-to-table mapping.
Deliver cabinets sequentially based on assignment order.
3.3 Robot Delivery Logic 
FR-12: Cabinet Allocation Rules
When assigning cabinets:
System shall:
Display only available cabinets.
Prevent assigning occupied cabinets.
Allow selecting multiple cabinets for same order.
Automatically mark selected cabinets as â€œOccupiedâ€.
FR-13: Delivery Execution
Upon delivery:
Robot arrives at Table X.
Only cabinets assigned to Table X shall unlock.
Other cabinets remain locked.
After confirmation:
Cabinets are marked as â€œAvailableâ€.
If no cabinets remain occupied:
Robot status becomes â€œIdleâ€.
Else:
Robot continues to next assigned table.
FR-14: Multi-Stop Delivery State Management
Robot states:
Idle
Delivering (Multi-stop)
Delivering (Final Stop)
Returning (Optional future enhancement)
System shall maintain:
Delivery queue per robot.
Table order sequence.
Cabinet-table mapping.

4. Non-Functional Requirements
4.1 Performance Requirements
System shall support minimum 100 concurrent users.
Order updates should reflect within 5 seconds.
4.2 Security Requirements
Role-based authentication.
Payment transactions must use secure HTTPS.
JWT-based authentication for APIs.
4.3 Usability Requirements
Mobile-responsive web interface.
Simple UI for staff and customers.
Owner dashboard optimized for mobile screens.
4.4 Reliability Requirements
System uptime target: 99%.
Order data shall not be lost during server restart.
4.5 Scalability
System shall support addition of multiple restaurants.
Database shall support horizontal scaling.
